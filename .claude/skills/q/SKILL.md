# quote-calc — 仕入原価計算・見積もり作成支援

旅行手配（特にホテル）の料金計算ミスを防ぐためのスキル。

## 起動ルール

- **このスキルは `/q` コマンドで明示的に呼び出された場合のみ起動する**
- 会話中に「見積もり」「原価」「料金計算」等のキーワードが出ても自動起動しない
- `/q` の後にカスタム指示を自由に追加できる（例: `/q calc マークアップ15%で計算して`）
- 見積もり作成を開始した後のスレッド内での追加指示・修正依頼は `/` なしでそのまま送信してOK

## コマンド

| コマンド | 用途 |
|---------|------|
| `/q calc [指示]` | ホテル見積もりから仕入原価・販売価格を計算。指示でカスタマイズ可 |
| `/q check` | 既存の見積もりデータの整合性チェック |
| `/q template` | 新規見積もり計算用のテンプレートを出力 |

---

## 1. ホテル料金計算の基本ルール

### 1.1 料金表記パターン（ホテル側からの提示）

ホテルから提示される料金は以下のパターンがある。**必ずどのパターンかを最初に確認すること**。

| パターン | 意味 | 例 |
|---------|------|-----|
| **税込・サ込** | 消費税もサービス料も込み | ¥30,000（税サ込） |
| **税込・サ別** | 消費税は込み、サービス料は別途 | ¥28,000（税込）+ サービス料10% |
| **税別・サ込** | サービス料は込み、消費税は別途 | ¥27,273（税別・サ込） |
| **税別・サ別** | 両方別途 | ¥25,000（税別）+ サービス料10% |

### 1.2 単位パターン

| パターン | 記載例 | 注意点 |
|---------|--------|--------|
| **1室1泊** | ¥30,000/室/泊 | そのまま泊数をかける |
| **1名1泊** | ¥15,000/名/泊 | 人数 × 泊数が必要 |
| **1室トータル** | ¥90,000/室（3泊） | 泊数で割って単泊チェック |
| **1名トータル** | ¥45,000/名（3泊） | 人数・泊数で割って単泊チェック |

### 1.3 消費税率

- **宿泊料**: 10%
- **食事（ルームサービス含む）**: 10%（ケータリング扱い）
- ※テイクアウト食品が分離されている場合は 8%（軽減税率）だが、ホテル見積もりでは通常10%

## 2. 仕入原価の計算手順

### ステップ1: ホテル提示料金を「税別・サ別」のベースに統一する

```
■ パターン別の逆算

【税込・サ込】→ 税別・サ別ベース
  base = 提示額 ÷ (1 + サービス料率) ÷ (1 + 税率)

【税込・サ別】→ 税別・サ別ベース
  base = 提示額 ÷ (1 + 税率)
  ※サービス料は別途加算されるので、baseがそのまま税別・サ別

【税別・サ込】→ 税別・サ別ベース
  base = 提示額 ÷ (1 + サービス料率)

【税別・サ別】→ そのまま
  base = 提示額
```

### ステップ2: コミッション（手数料）の処理

ホテル側がコミッション付きで提示する場合：

```
■ コミッションの計算基礎を確認すること！

パターンA: 税別・サ別ベースに対してコミッション
  仕入原価(税別) = base × (1 + サービス料率) × (1 - コミッション率)
  仕入原価(税込) = 仕入原価(税別) × (1 + 税率)

パターンB: 税サ込金額に対してコミッション
  支払総額(税サ込) = base × (1 + サービス料率) × (1 + 税率)
  コミッション額 = 支払総額 × コミッション率
  仕入原価(税込) = 支払総額 - コミッション額

パターンC: コミッションなし（ネットレート）
  仕入原価(税別) = base × (1 + サービス料率)
  仕入原価(税込) = 仕入原価(税別) × (1 + 税率)
```

**⚠️ 重要: コミッションの計算基礎（税前 or 税サ込）はホテルごとに異なる。必ず確認すること。**

### ステップ3: 総額計算

```
仕入原価合計 = 単泊仕入原価 × 泊数 × 室数
               ※1名料金の場合: × 人数 も必要
```

---

## 3. 販売価格の算出

### 3.1 マークアップ方式

```
販売価格 = 仕入原価(税込) × (1 + マークアップ率)
利益 = 販売価格 - 仕入原価(税込)
利益率 = 利益 ÷ 販売価格 × 100
```

### 3.2 利益率から逆算

```
販売価格 = 仕入原価(税込) ÷ (1 - 目標利益率)
```

## 4. 計算実行時のルール

### `/q calc` 実行時

ユーザーからホテル見積もり情報を受け取ったら、以下の手順で処理する：

1. **情報の確認・整理**
   - ホテル名
   - 提示料金と表記パターン（税込/税別、サ込/サ別）
   - 単位（1室 or 1名、1泊 or トータル）
   - サービス料率（通常10%、ホテルにより異なる）
   - コミッション率とその計算基礎
   - 宿泊日数、室数、人数

2. **不明点があれば必ず確認する**
   - 「この料金は税込ですか？税別ですか？」
   - 「サービス料は込みですか？別途ですか？」
   - 「コミッションの計算基礎は税前？税サ込？」
   - 「1名料金ですか？1室料金ですか？」

3. **計算結果を以下のフォーマットで出力する**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🏨 ホテル名: ○○○○
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【提示条件】
  料金表記: 税別・サ別
  単位: 1室1泊
  サービス料: 10%
  コミッション: 10%（税別ベース）

【計算過程】
  ① ベース料金（税別・サ別）: ¥25,000
  ② サービス料加算: ¥25,000 × 1.10 = ¥27,500
  ③ コミッション控除: ¥27,500 × (1 - 0.10) = ¥24,750
  ④ 消費税加算: ¥24,750 × 1.10 = ¥27,225
  → 1室1泊の仕入原価（税込）: ¥27,225

【総額】
  泊数: 3泊 × 1室
  仕入原価合計（税込）: ¥27,225 × 3 = ¥81,675

【販売価格（マークアップ20%の場合）】
  販売価格: ¥81,675 × 1.20 = ¥98,010
  利益: ¥16,335
  利益率: 16.7%
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

4. **クロスチェック**
   - 計算結果を逆算して元のホテル提示額と一致するか検算
   - 1泊あたり単価が相場と大きく乖離していないか確認

### `/q check` 実行時

既存の見積もりデータを読み込み、以下をチェック：
- 税込/税別の一貫性
- 掛け算の正確性（泊数 × 室数 × 人数）
- コミッション計算の正確性
- 合計金額の整合性

## 5. よくあるミスと注意点

### 絶対にやってはいけないこと

| ミス | 何が起きるか |
|------|-------------|
| 税込料金にさらに消費税をかける | 原価が10%膨らむ → 赤字 |
| 1名料金を1室料金と間違える | 人数分の掛け算漏れ → 大幅赤字 |
| コミッションの基礎を間違える | 利益計算がずれる |
| サービス料の二重計上 | 原価が膨らむ |
| 泊数の数え間違い（チェックイン/アウト日） | 1泊分のずれ |

### チェックリスト（計算前に必ず確認）

- [ ] ホテル提示料金の税込/税別を確認したか？
- [ ] サービス料が込みか別途か確認したか？
- [ ] 1名料金か1室料金か確認したか？
- [ ] コミッションの計算基礎を確認したか？
- [ ] 泊数はチェックイン〜チェックアウトで正しく数えたか？
- [ ] 食事付きプランの場合、食事代が含まれているか確認したか？

## 6. 送迎・アクティビティ等の計算

ホテル以外の手配項目は比較的シンプル：

```
【送迎】
  仕入原価 = 業者提示額（通常税込）
  販売価格 = 仕入原価 × (1 + マークアップ率)

【アクティビティ・観光】
  仕入原価 = 業者提示額（税込/税別を確認）
  販売価格 = 仕入原価 × (1 + マークアップ率)

【レンタカー】
  手配料のみ請求（車両代はお客様現地払い）
  手配料 = 固定額（例: ¥15,000）
```

## 8. 見積書・旅程表の画像ハンドリング

HTML形式の見積書・旅程表を作成する際の画像に関するルール。

### 8.1 最優先: ユーザー指定画像

ユーザーが画像ファイルを指定した場合（パスやファイル名を提示した場合）、**必ずその画像を使用すること**。別の画像を検索して差し替えてはならない。

**手順:**
1. `Read` ツールで画像ファイルを読み込む（画像の内容を確認）
2. `Bash` で base64 エンコードする:
   ```bash
   base64 -i "/path/to/image.jpg" | tr -d '\n'
   ```
3. HTML に data URI として埋め込む:
   ```html
   <img src="data:image/jpeg;base64,{base64データ}" alt="説明">
   ```
4. ファイルサイズが大きすぎる場合（1MB超）は、HTMLと同じディレクトリに画像をコピーし、相対パスで参照する:
   ```bash
   cp "/path/to/image.jpg" "/output/dir/images/filename.jpg"
   ```
   ```html
   <img src="images/filename.jpg" alt="説明">
   ```

**重要:** ユーザーが「この写真を使って」「この画像を入れて」と指示した場合、その画像を忠実に使用する。勝手に別の画像に差し替えない。

### 8.2 画像を自動取得する場合の品質ルール

ユーザーが画像を指定していない場合に限り、以下のルールで画像を探す。

**禁止画像:**
- 文字情報（テキスト、ロゴ、キャプション、ウォーターマーク）が含まれている画像
- 販促バナーや広告素材
- 低解像度・ぼやけた画像
- スクリーンショット

**適切な画像:**
- ホテルの部屋・外観・ロビーの写真（文字なし）
- 観光地・風景の写真（文字なし）
- 車両の実物写真（文字なし）

### 8.3 画像取得の優先順位

1. **ユーザー指定画像**（最優先 — 必ずそのまま使う）
2. **WebSearch + WebFetch** で公式サイトや予約サイトから画像URLを取得
   - WebSearch でホテル名・施設名を検索
   - 検索結果のURLを WebFetch で取得し、画像URLを抽出
   - 取得した画像URLを `<img src="URL">` でHTMLに埋め込む
3. **browser-use** でホテル公式サイトや予約サイトにアクセスして画像を取得
   - ヘッドレス → ブロック時はリアルブラウザにフォールバック
4. **プレースホルダー** — 適切な画像が見つからない場合:
   ```html
   <div style="width:46mm;height:32mm;background:#f0f0f0;display:flex;align-items:center;justify-content:center;color:#999;font-size:8pt;border-radius:4px;">
     写真: ホテル名
   </div>
   ```
   ユーザーに「画像が取得できなかったため、後から差し替えてください」と報告する。

### 8.4 画像選定時の確認ポイント

画像を選ぶ際は、以下を自問すること:
- この画像に文字・テキストが含まれていないか？ → 含まれていたら **使わない**
- ユーザーが指定した画像と異なるものに差し替えていないか？ → 差し替えていたら **やり直す**
- 画像の解像度は十分か？（最低でも 400px 以上の幅）

## 9. セクション管理（HTML見積書・旅程表の編集ワークフロー）

### 9.1 セクションID体系

HTML見積書・旅程表を生成する際、各セクションに `id` 属性を付与し、ユーザーがセクション単位で編集・削除できるようにする。

**標準セクション一覧:**

| セクションID | セクション名 | 内容 |
|---|---|---|
| `sec-accommodation` | 宿泊施設 | ホテルカード群 |
| `sec-vehicle` | レンタカー / 車両 | レンタカー・送迎車両カード |
| `sec-arrangement` | 手配内容 | 手配状況テーブル（誰が手配・誰が支払い） |
| `sec-pricing` | 見積金額 | 金額サマリー（pbox） |
| `sec-payment` | 支払い方法 | 決済手段の説明 |
| `sec-cancel-policy` | キャンセルポリシー | キャンセル料テーブル |
| `sec-insurance` | 保険 | 旅行保険の案内 |
| `sec-itinerary` | 旅程表 | 日程ごとの行程（DAY 1, DAY 2...） |
| `sec-notes` | 備考 | 注意事項・免責事項 |
| `sec-transfer` | 送迎 | 空港送迎等 |
| `sec-activity` | アクティビティ | オプショナルツアー等 |

### 9.2 HTMLでのセクションマークアップ

各セクションを `<section id="sec-xxx">` で囲む:

```html
<!-- セクション: 宿泊施設 -->
<section id="sec-accommodation">
  <div class="sec-title">宿泊施設</div>
  <!-- ホテルカード群 -->
</section>

<!-- セクション: 手配内容 -->
<section id="sec-arrangement">
  <div class="sec-title">手配状況</div>
  <!-- 手配テーブル -->
</section>

<!-- セクション: 見積金額 -->
<section id="sec-pricing">
  <div class="pbox">
    <!-- 金額サマリー -->
  </div>
</section>

<!-- セクション: キャンセルポリシー -->
<section id="sec-cancel-policy">
  <div class="sec-title">キャンセルポリシー</div>
  <!-- キャンセル料テーブル -->
</section>
```

### 9.3 生成後のセクションサマリー出力（必須）

HTML見積書・旅程表を生成した後、**必ず**以下の形式でセクション構成をユーザーに提示する:

```
📄 見積書を作成しました → [ファイルパス]

【セクション構成】
1. 宿泊施設 — MIMARU東京池袋(6泊)、富士レークホテル(3泊)、メトロポリタン鎌倉(4泊)
2. レンタカー — ヴォクシー ウェルキャブ（スロープタイプ）
3. 手配内容 — 各項目の手配者・支払者一覧
4. 見積金額 — 合計 ¥1,634,892（税込）
5. 支払い方法 — FlyWire決済
6. キャンセルポリシー — 14日前25%〜当日100%
7. 旅程表 — DAY1〜DAY14
8. 備考 — 為替変動・料金改定に関する注意事項

修正があれば指示してください。例:
  - 「手配内容、削除して」
  - 「キャンセルポリシー、30日前から無料に変更して」
  - 「保険セクションを追加して」
  - 「旅程表のDAY3を変更して」
```

### 9.4 セクション編集のルール

ユーザーからセクションに関する指示を受けた場合、**必ずその通りに実行する**。

#### 削除指示の場合

ユーザーが「○○を削除」「○○いらない」「○○消して」と言った場合:
1. 該当セクションの `<section id="sec-xxx">...</section>` ブロック全体を `Edit` ツールで削除する
2. 削除後、残ったセクション構成を再度提示する
3. **絶対に「削除しない方がいい」等の判断を勝手にしない。ユーザーの指示通りに削除する。**

#### 変更指示の場合

ユーザーが「○○をこう変更して」と言った場合:
1. 該当セクションの内容を `Edit` ツールで指示通りに修正する
2. 変更後の内容を簡潔に報告する

#### 追加指示の場合

ユーザーが「○○セクションを追加して」と言った場合:
1. 適切な位置に新しい `<section id="sec-xxx">` を `Edit` ツールで追加する
2. 追加後のセクション構成を再度提示する

### 9.5 セクション操作の鉄則

- **ユーザーが「削除」と言ったら削除する。** 理由を聞いたり代替案を提案したりしない。
- **ユーザーが「変更」と言ったら変更する。** 元の内容を残さない。
- **セクション操作はHTMLファイルの `Edit` で行う。** ファイル全体を書き直さない。
- **操作後は必ず更新されたセクション構成を提示する。**

## 10. 言語統一ルール（厳守）

見積書・旅程表のHTMLを生成する際、**1つのドキュメント内で言語を完全に統一する**。混在は絶対に許されない。

### 10.1 対応言語と決定方法

対応言語は **日本語・韓国語・英語・中国語（繁体字）** の4言語。

ユーザーの指示または顧客の言語に基づいて、ドキュメント全体の言語を1つに決定する。
**決定した言語以外のテキストが1文字でもHTMLに含まれてはならない**（固有名詞・ブランド名「Flat Travel」「gotcha」「tabiho」等を除く）。

`<html lang>` 属性も必ず設定: `ja` / `ko` / `en` / `zh-Hant`

### 10.2 生成プロセス（日本語テンプレートから翻訳するな）

**⚠️ 日本語のHTMLを書いてから翻訳する方式は禁止。**

この方式だと翻訳漏れが必ず発生する（PDFで実証済み）。代わりに以下のプロセスで生成する:

#### ステップ1: テキスト要素の全量洗い出し

HTMLを書き始める前に、ドキュメントに含まれる**すべてのテキスト要素**を洗い出す:

- ドキュメントタイトル
- 各セクションタイトル
- 基本情報のラベル（旅行期間、顧客名、人数、航空券等）
- 基本情報の値
- テーブルのヘッダー（すべてのテーブル）
- テーブルのセル内容
- バッジ・ステータスラベル
- 金額ラベル・通貨表記
- 税込/税別表記
- 日付（フォーマット含む）
- 曜日
- 食事ラベル
- タグ（部屋タイプ、サービス等）
- カード内の説明文
- 注意書き・備考
- 保険案内の本文
- 会社情報
- ページヘッダーの日付

**1つでも見落としたら言語混在の原因になる。**

#### ステップ2: 全要素を指定言語で記述

洗い出した全要素を、最初から指定言語で書く。日本語で書いてから置換するのではなく、最初から韓国語なら韓国語で、英語なら英語で書く。

#### ステップ3: HTMLを組み立てる

ステップ2で用意したテキストを使ってHTMLを組み立てる。

### 10.3 セルフチェック（生成後に必ず実行）

HTMLを生成した後、**Writeする前に**以下のチェックを行う:

#### 韓国語・英語・中国語ドキュメントの場合

生成したHTML全体を読み返し、**日本語の文字（ひらがな・カタカナ・漢字）が残っていないか確認**する。

具体的には、以下のような文字列が残っていたら言語混在エラー:
- ひらがな: お、ご、の、は、で、に、を、が、と、も、へ、から、まで、です、ます
- カタカナ: キャンセル、チャーター、レンタカー、アクティビティ
- 漢字ラベル: 旅行期間、手配、支払、項目、詳細、合計、税込、弊社、現地、保険、備考、旅程表、期間、航空券、宿泊

**該当箇所が1つでも見つかったら、指定言語に書き直してからWriteする。**

#### 英語・中国語ドキュメントで韓国語が混ざっていないかも確認

同様に、韓国語の文字（ハングル）が英語・中国語ドキュメントに混ざっていないか確認する。

**このチェックをスキップしてはならない。チェックなしでWriteすることは禁止。**
